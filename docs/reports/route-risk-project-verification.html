Instructor presentation

Recommended demonstration walkthrough

This order begins with the working product, exposes the distributed architecture, and ends with quantitative evidence.

  1. Introduce the problem and architecture. Explain that the system evaluates route safety rather than only travel time, using FastAPI, Redis, Celery workers, weather data, roadway events, scoring, and aggregation.
  2. Select the origin and destination. Click two points on the dashboard map and submit the analysis.
  3. Show distributed processing. Briefly point to active worker logs while route checkpoints are processed.
  4. Review the three route alternatives. Show actual road geometry, distance, estimated time, checkpoint counts, risk score, and road-event information.
  5. Explain the recommendation. Show that blocked routes are rejected first, then lower-risk routes are prioritized before time and distance.
  6. Demonstrate closure behavior. Point out a route closure or construction result and explain why a closure cannot be averaged into an otherwise low score.
  7. Open the final benchmark page. Review the 40-task workload, 1/2/4/8-worker comparison, 7.35× speedup, and the three generated graphs.
  8. Close with the learning outcome. Summarize what was learned about distributed task queues, integration, fault-aware scoring, aggregation, Docker, and performance measurement.

Presentation note

Do not turn this into another application.

This page is a walkthrough and evidence checklist. The existing route dashboard remains the product, and the README remains the technical installation and project documentation.

Keeping these roles separate avoids duplicating dashboard code and makes the final presentation easier to follow.
